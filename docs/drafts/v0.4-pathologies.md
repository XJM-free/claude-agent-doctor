# v0.4 pathology drafts

> **Status: drafts, not implemented.** This file documents the next four
> pathologies planned for v0.4. It exists so the roadmap is visible, the
> mechanisms are public, and contributors can propose PRs implementing any of
> them independently.
>
> Each draft below has a detection signature, the fields it would need on
> `SessionStat` (if any), and an expected mechanism. When implemented, the
> entry graduates to `PATHOLOGY.md`.

## `MCP_CHATTER` (tools, med)

### Summary (draft)
A single MCP tool accounts for >20% of a session's tool_use invocations — indicates an over-chatty MCP integration or a tight loop driven by one server.

### Mechanism
MCP servers are useful but expensive: every call is a round-trip to an external process, and the tool-use payload is serialized back into context. When one MCP tool dominates, you're usually either (a) using a "swiss-army" server where a purpose-built call would suffice, or (b) the agent is polling the server in a loop.

### Detection signature
```
for each tool with name starting "mcp__":
  if count / total_tool_count >= 0.20 AND count >= 20:
    hit
```

### Required SessionStat additions
None — current `toolCount` already keys by name including `mcp__*` prefix.

### Prescription
- Look at `doctor check <session-id>` to see which MCP dominates.
- If that server exposes higher-level actions, prefer them over fine-grained ones.
- For polling-style servers, consider switching to a subscription model (e.g. MCP resources) if the server supports it.

## `TEST_BYPASS` (tools, med)

### Summary (draft)
5+ edits in a session without a single `Bash` invocation that looks like a test or build run — strong signal the agent is shipping without verification.

### Mechanism
In healthy workflows, Edit-heavy sessions include regular `Bash` calls to `cargo test`, `bun test`, `xcodebuild`, `npm run lint`, `python -m pytest`, etc. When many edits happen without any such calls, the agent is either confident in purely-mechanical changes (OK) or skipping verification (risky). The rule is permissive: it only fires when the edit count is meaningful.

### Detection signature
```
sum(editedFiles) >= 5
AND no Bash invocation matches:
  /^(cargo|bun|npm|pnpm|yarn|python|xcodebuild|swift|go)\s+(test|lint|typecheck|build|run)/
```

### Required SessionStat additions
Track a flag or counter for "verification-like Bash calls". Simplest: add
`verificationBashCount: number` populated during `parseSession` by regex on
the Bash `command` input.

### Prescription
- Run tests after edit-heavy sessions; if the tests are slow, pin the critical test subset.
- Add a CLAUDE.md note: "after any edit larger than 20 lines, run the relevant tests before declaring done."

## `SYSPROMPT_BLOAT` (tools, low)

### Summary (draft)
First-turn input exceeds 20K tokens — the system prompt + CLAUDE.md + agent config has gotten obese.

### Mechanism
Claude Code assembles each turn's input from: system prompt, CLAUDE.md (project + user + global), enabled skills' descriptions, MCP tool definitions, and any prior conversation. When the first turn of a session already tops 20K tokens, the stack has gotten heavy. This is the "tax" you pay on every single subsequent turn through cache reads.

### Detection signature
```
first_turn_total_input_tokens >= 20_000
```

### Required SessionStat additions
`firstTurnInputTokens: number` — the `totalInputTokens` value of the first assistant turn only.

### Prescription
- Audit CLAUDE.md for dead sections (outdated rules, duplicate content across scopes).
- Prefer scoping rules to specific projects rather than global.
- Check enabled Skills — each description stays in context at all times.

## `DEAD_SUBAGENT` (config, low)

### Summary (draft)
An agent file under `~/.claude/agents/` exists but was never invoked in the observed window — dead config adding description-text tax to every session.

### Mechanism
Every subagent's description lives in context so the orchestrator knows when to route to it. Agents that never fire are still paying that tax on every session, forever. This is less a runtime issue and more a config hygiene one, hence `severity: low`.

### Detection signature
```
for each agent file found by findAllAgentFiles():
  if agent.name not seen in any Agent/Task tool_use in window:
    hit
```

### Required SessionStat additions
`subagentInvocations: Record<string, number>` — per-subagent-type call count. The JSONL reader currently throws this away (known TODO in `routing.ts::countSubagentInvocations`).

### Prescription
- Move truly-dead agents out of `~/.claude/agents/` (to a `~/.claude/agents/.archive/` directory).
- For agents used rarely, consider pinning them to `disable-model-invocation: true` so they only run when you `/<name>` them explicitly.

## Implementation notes

These four expand the catalog from 12 → 16 pathologies and together require:
- `verificationBashCount` on SessionStat (TEST_BYPASS)
- `firstTurnInputTokens` on SessionStat (SYSPROMPT_BLOAT)
- `subagentInvocations` on SessionStatsBundle (DEAD_SUBAGENT)

`MCP_CHATTER` needs no new fields — implemented as a detector over existing `toolCount`.

See `CONTRIBUTING.md` for the bar a pathology must clear to graduate from draft to PATHOLOGY.md.
